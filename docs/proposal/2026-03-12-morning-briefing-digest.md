# Morning Briefing (通知ダイジェスト機能)

## Summary
夜間など指定期間内に実行された複数タスクの実行結果、生成されたメモリ、およびエラーログを、指定した時刻（例：毎朝8時）に LLM が自動で要約し、1つの読みやすいメッセージとして Slack や Discord に一括送信する「Morning Briefing」機能の提案。

## Problem
現在、`kage` のコネクター（Slack, Discord 等）はタスクの完了やエラーごとに個別の通知を送信します。
夜間に多数の自動化タスク（例：複数リポジトリの監査、複数モデルのベンチマーク、ニュース収集など）を実行している場合、朝起きるとチャットツールに大量の通知が並んでしまい、重要な結果が埋もれてしまう（通知のノイズ化）という問題があります。
毎日運用する個人用エージェントとして、朝一番の認知負荷を下げる必要があります。

## Proposal
`.kage/config.toml` または専用のタスク設定として、通知をバッチ化して要約する `digest` 機能を追加します。

```toml
[notifications.digest]
enabled = true
schedule = "0 8 * * *" # 毎朝8時に送信
provider = "claude"
prompt = "過去24時間のkageタスク実行結果を要約し、注目すべき発見や失敗したタスクを中心に報告してください。"
```

内部動作:
1. `kage` は各タスクの完了通知を即時送信せず（または即時送信と併用で）内部の SQLite (`kage.db`) に結果とメモリの差分をキューイングする。
2. スケジュールされた時刻に、未報告のタスク実行履歴・メモリ差分・エラー内容をコンテキストとして LLM に渡し、プロンプトに従ってダイジェストを生成。
3. 生成された要約テキストをコネクター経由で送信する。

## Inspiration
- **Slack のデイリーサマリー機能 / GitHub のデイリーダイジェスト**: ユーザーの関心を惹くトピックだけをまとめて朝に送る UX。
- **Claude / ChatGPT の長文要約機能**: 構造化されていない雑多なログを、人間が読みやすいインサイトに変換する能力。

## Expected Daily Benefit
- 朝起きたときに、1つのメッセージを読むだけで夜間の AI エージェントの全活動と成果を素早く把握できるようになり、一日のスタートがスムーズになる。
- 通知疲れ（アラート疲労）が解消され、重要なエラーやインサイトを見落とさなくなる。
- 複数の `kage` プロジェクト（複数ディレクトリ）をまたいだ Global Dashboard の情報も、通知として手軽に受け取れるようになる。

## Scope and Non-Goals
- **Scope**: SQLite 内の実行履歴と最近の `.kage/memory/` の変更分を LLM に渡して要約させ、設定されたコネクターに送信する処理。
- **Non-Goals**: リアルタイムな緊急アラート（エラー時など）の抑制。致命的なエラーについては即時通知を維持する（`priority` などの概念導入は今回はスコープ外、または別の提案とする）。

## Risks or Open Questions
- **コンテキストウィンドウの枯渇**: 多数のタスクの出力を LLM に渡すとトークン長を超過する可能性がある。各タスクの出力を事前に切り詰める（Truncation）か、重要度の高いものだけを抽出するロジックが必要。
- **LLM のコスト**: 毎日の要約に高価なモデル（Claude 3.5 Sonnet など）を使うとコストが嵩むため、サマリー用途には安価なモデル（Haiku, Gemini Flash など）をデフォルトで指定できるようにするべきか。

## Suggested Rollout
1. まず CLI コマンドとして `kage digest --since "24 hours ago"` を実装し、標準出力にマークダウンの要約を出す機能を先行リリースする。
2. 次に、`config.toml` でのスケジュール（cron）設定と、Slack/Discord への送信ロジックを統合する。
3. 即時通知をオフにする（ダイジェストのみにする）タスクごとのフラグ（例: `notify: digest_only`）をフロントマターに追加する。
