# タスク実行のタイムアウトと強制終了 (Execution Timeout)

## Summary

タスクのフロントマターに `timeout_minutes` を設定可能にし、指定時間を超えても終了しないタスク（およびその子プロセス）を強制終了してエラーとして記録・通知する機能の提案。

## Problem

`kage` の中核価値は「夜間に自律稼働させ、朝に結果を得る」ことですが、現在の仕様ではタスクの実行時間に関する制限がありません。
エージェントが生成したスクリプトが無限ループに陥ったり、`curl` や外部 API 呼び出しがネットワークの不具合でハングアップした場合、タスクプロセスは永遠に終了せず残り続けます。
これにより、「後続の cron タスクが concurrency 制限でブロックされる」「PC の CPU やメモリリソースを夜通し食いつぶす」「朝起きた時に何も成果物が出ていない（サイレント障害）」という、自動化における最悪の DX 低下を引き起こします。

## Proposal

タスク定義 (`.kage/tasks/*.md`) のフロントマターにおいて、タスク全体の最大実行時間を制限する `timeout_minutes` フィールドを追加します。

1. **フロントマターでの指定**:
   ```yaml
   name: "Heavy Data Processing"
   cron: "0 2 * * *"
   timeout_minutes: 60
   ```
2. **タイムアウト時の強制終了**:
   タスクの実行開始時に監視タイマーを起動します。指定時間を超過した場合、まずプロセス（およびそのプロセスグループ）に対して `SIGTERM` を送信してグレースフルな終了を促し、数秒待っても終了しない場合は `SIGKILL` で強制終了（Hard Kill）します。
3. **状態の記録と通知**:
   タイムアウトによって強制終了されたタスクは、`.kage/memory/` に `status: failed (timeout after 60 minutes)` として記録されます。Connector（Slack / Discord 等）が設定されている場合は、通常の失敗と同様にタイムアウト通知を送信します。

## Inspiration

- **GitHub Actions**: `timeout-minutes` パラメータ。CI/CD ジョブがハングアップしてクレジットを浪費するのを防ぐ標準的な安全装置。
- **Linux `timeout` コマンド**: 特定のコマンドを指定時間で打ち切る仕組み。
- **systemd / launchd**: サービスの `TimeoutStartSec` や `TimeoutStopSec`。

## Expected Daily Benefit

エージェントの予測不可能な出力（バグのあるコードなど）や、外部依存システムの不安定さに起因する「ハングアップによる一晩の無駄」を完全に防ぎます（Safe-guard）。
タスクがハングアップした場合でも、永遠に待ち続けるのではなく明確に「失敗（Timeout）」として着地するため、朝起きた時に「なぜ止まっていたのか」を正確に把握でき、夜間バッチに対する安心感と信頼性が大きく向上します。

## Scope and Non-Goals

- **Scope**: タスクの実行全体（LLM 思考時間、シェルコマンド実行時間を含むトータル）に対するタイムアウト監視と、プロセスグループ全体の確実な終了（Orphan プロセスを残さない処理）。
- **Non-Goals**: LLM API の 1 回の呼び出しごとの細かいタイムアウト設定（それは AI クライアントライブラリ側の責務）や、特定のステップ（サブタスク）単位でのタイムアウトはスコープ外とし、あくまで「タスク全体」の制限に特化します。

## Risks or Open Questions

- **プロセスグループの終了**: エージェントがシェル経由でバックグラウンドプロセスを起動していた場合、単純に kage のメインプロセスを kill するだけでは子プロセス（ゾンビプロセス）が残る可能性があります。`os.setsid()` 等を用いたプロセスグループ（PGID）単位での確実なクリーンアップ実装が必要です。
- **状態の不整合**: 強制終了された瞬間に SQLite (`kage.db`) やメモリファイルへの書き込みが行われていた場合、ファイルが破損するリスクがあります。タイムアウト処理側で安全に状態を記録できるトランザクション設計が求められます。

## Suggested Rollout

1. **Phase 1**: フロントマターでの `timeout_minutes` のパースと、タスク実行ループにおけるシンプルなタイマー監視の実装（Python の `asyncio.wait_for` や別スレッドでの監視）。
2. **Phase 2**: タイムアウト検知時の `SIGTERM` -> `SIGKILL` 送信ロジック（プロセスグループへのシグナル送信）の追加。
3. **Phase 3**: タイムアウトによる強制終了時に、データベースやファイルシステムの状態を安全にクリーンアップし、Connector へ通知を送るエラーハンドリングの整備。
