# `kage tail`: 実行中タスクのリアルタイムログストリーミング (Live Execution Streaming)

## Summary
現在ヘッドレスでバックグラウンド実行されている `kage` のタスクに対し、CLI から `kage tail <task_name>` コマンドで接続し、エージェントの思考プロセス (`<think>`) や実行しているコマンドの標準出力・標準エラーをリアルタイムで監視（ストリーミング）できる機能の提案。

## Problem
`kage` は `cron` や `launchd` によるヘッドレスなバックグラウンド実行を基本としているため、「寝ている間に完了する」用途には最適です。しかし、「今まさに動いているタスクがどこまで進んでいるか」「AIが無限ループや無駄な操作に陥っていないか」を確認したい場合、ダッシュボードや静的なログファイルのリロードに頼る必要があり、DX（開発体験）やデバッグ性が高くありませんでした。

## Proposal
- **`kage tail <task_name>` コマンドの追加**: 実行中の該当タスクプロセスを特定し、そのログ（標準出力/標準エラー）を現在のターミナルにアタッチしてリアルタイムでストリーミング表示します。
- **思考プロセスの可視化**: デフォルトでは `<think>` タグ内などエージェントの内部推論もストリーミングし、デバッグを容易にします。オプション（`--no-think` など）で結果のみをフィルタリングできるようにします。
- **読み取り専用アタッチ**: あくまでログの監視であり、ストリーミングプロセスを `Ctrl+C` で終了しても、バックグラウンドの `kage` タスク本体の実行は継続するようにします。

## Inspiration
- `docker logs -f <container>` や `kubectl logs -f <pod>` のコンテナログ監視体験。
- `tmux` の読み取り専用アタッチモード。
- CI サービス（GitHub Actions）のリアルタイムコンソールログ。

## Expected Daily Benefit
- エージェントが期待通りの思考ルートを辿っているかを**実行中に即座に確認**できるため、タスク定義（プロンプト）の調整サイクルが圧倒的に早くなります。
- 重い処理や長時間のビルドがハングしていないかを直感的にモニタリングできるようになります。

## Scope and Non-Goals
- **Non-Goal**: `kage tail` 画面から対話的にエージェントに指示を与えたり、プロンプトを再入力する機能（Interactive mode）はスコープ外です。あくまで「監視」に留め、介入（中断）が必要な場合は `kage kill` やタスク設定の変更で対応します。

## Risks or Open Questions
- **ログの書き込み・同期**: 既存の実装が標準出力を単一のファイルや DB にバッファリングしている場合、`tail` 側でリアルタイム性が損なわれる可能性があります。ファイルディスクリプタのフラッシュや専用のログアーキテクチャへの変更が必要になるかもしれません。
- **実行プロセスの特定**: タスク名から OS レベルの PID や該当のテンポラリログファイルを一意かつ安全に特定する仕組みが必要です。

## Suggested Rollout
1. 最初にタスクの標準出力・エラー出力をタスク専用のテンポラリログファイル（例: `.kage/logs/<task_name>_<run_id>.log`）にラインバッファ・追記モードで書き出す基盤を整備する。
2. `kage tail <task_name>` が該当タスクの最も新しい実行中ログファイルを特定し、`tail -f` 相当の読み出し処理を行う機能を実装する。
3. （将来）Web ダッシュボード側でも同じログストリーミングを WebSocket 等で提供する。
