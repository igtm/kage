# タスク失敗時の自己修復 (Self-Healing / Auto-Correction) 機構の導入

## Summary

エージェントが生成したコードの実行や、指定されたシェルコマンドがエラー（非ゼロ終了）となった場合に、そのエラーログをプロンプトにフィードバックして自律的に修正・再実行を試みる「自己修復（Self-Healing）」機能の提案。

## Problem

`kage` の最大の価値は「Go to sleep. Wake up to results.（寝ている間に結果を出す）」ですが、現在の仕様ではエージェントが生成したコードに軽微なシンタックスエラーがあったり、パッケージのインストール忘れなどで実行が失敗した場合、タスクはそのまま異常終了してしまいます。
結果として、ユーザーは朝起きたときに「完了した成果物」ではなく「途中で止まったエラーログ」を見ることになり、手動でエラーを修正して再度タスクを起動しなければならないという体験の劣化（DXの低下）が生じています。

## Proposal

タスクのフロントマターに `auto_heal: true` および `max_retries` の設定を追加し、エラー発生時の自律的なリトライループをサポートします。

1. **エラーの捕捉**:
   タスク内のスクリプト実行やシェルコマンド（フロントマターの `command` を含む）が非ゼロ終了（Exit Code != 0）した場合、標準出力（stdout）と標準エラー出力（stderr）をキャプチャします。
2. **自己修復プロンプトの自動生成**:
   キャプチャしたエラーログを含めた「修正要求プロンプト」（例: `The previous execution failed with the following error:\n\n{error_log}\n\nPlease analyze the error and provide a corrected implementation.`）を自動生成し、AI のコンテキストに追加します。
3. **自律的な再実行 (リトライ)**:
   ユーザーの介入を待つことなく、即座に（または次回の cron ティックで）AI に修正版を生成させ、再度実行を試みます。このループをフロントマターで指定された `max_retries`（デフォルト3回など）に達するまで繰り返します。

## Inspiration

- **aider / OpenHands / Claude Code**: テストが落ちたりコマンドが失敗した際に、エラー出力を読み込んで勝手に自己修正する強力な体験。
- **AutoGPT / BabyAGI**: アクションの失敗をフィードバックとして受け取り、次の戦略を練り直すループ構造。
- **CI/CD の flaky test リトライ**: 失敗したステップを自動で再実行する仕組みのAI版。

## Expected Daily Benefit

「一発で完璧なコードを書けなかった」という LLM の限界を、リトライループによってカバーできます。
軽微なタイポ、`pip install` の抜け漏れ、APIの一時的なタイムアウトなどの「人間が見れば数秒で直せるエラー」をエージェント自身が夜通し試行錯誤して乗り越えてくれるため、朝起きたときの「タスク完遂率」が飛躍的に向上します。

## Scope and Non-Goals

- **Scope**: コマンド実行の Exit Code を監視し、失敗時にエラーログをプロンプトに注入して再帰的に AI を呼び出す自己修復ループの実装。
- **Non-Goals**: 無限ループによる API コストの青天井を防ぐため、必ず `max_retries` による上限（Circuit Breaker）を設けます。また、論理的なバグ（エラーは出ないが結果が間違っている）の検知は難しいため、スコープ外（明示的なテストコードが失敗した場合のみ対象）とします。

## Risks or Open Questions

- **API コストの増加**: 失敗のたびにコンテキスト全体を送信し直すため、トークン消費が増大します。エラーログが長すぎる場合の自動切り詰め（Truncation）処理が必要です。
- **cron 実行との兼ね合い**: 失敗した場合に即座に同期的にリトライするのか、次回の cron 実行まで待つのか。同期的にリトライしたほうがコンテキストの鮮度や解決スピードは高いですが、kage プロセスの実行時間が長くなるというトレードオフがあります。

## Suggested Rollout

1. **Phase 1**: フロントマターでの `max_retries` と `auto_heal` フィールドのサポート定義。
2. **Phase 2**: コマンド実行レイヤーでの Exit Code 監視と stdout/stderr のキャプチャ、およびエラーログの自動切り詰めロジック（末尾 N 行の抽出など）の実装。
3. **Phase 3**: エラー捕捉時にプロンプトを構築し、再帰的に AI 実行ループを回す処理（Circuit Breaker 付き）の導入。
