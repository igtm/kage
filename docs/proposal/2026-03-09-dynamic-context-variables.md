# タスクプロンプトへの動的変数バインディング (Dynamic Context Variables)

## Summary

タスクの Markdown 本文内に `{{ date.today }}` や `{{ exec "git rev-parse HEAD" }}` のようなテンプレート変数を記述できるようにし、LLM へ送信する直前に `kage` 側で事前評価・置換を行う機能の提案。

## Problem

現在の `kage` のタスク定義は静的な Markdown です。そのため、「昨日のログを解析して」「最新のコミットの内容をレビューして」といった動的なコンテキストを扱う場合、エージェントは実行開始後に自律的に `date` コマンドや `git` コマンドを叩いて情報を収集するステップを踏む必要があります。
これは以下の問題を引き起こします：

1. **API コストと時間の浪費**: 情報収集のためだけの無駄な LLM ターン（思考とツール呼び出し）が発生する。
2. **不安定性**: エージェントが適切なコマンドを思いつけなかったり、コマンド実行に失敗すると、本題に入る前にタスクが頓挫する。

## Proposal

タスク定義 (`.kage/tasks/*.md`) の本文内で、簡易的なテンプレート構文（Jinja2 や Handlebars ライク）をサポートします。

1. **組み込み変数の提供**:
   - `{{ date.today }}` (YYYY-MM-DD 形式)
   - `{{ date.yesterday }}`
   - `{{ env.USER }}` などの環境変数
2. **コマンド実行結果の事前バインディング**:
   - `{{ exec "git diff origin/main --name-only" }}` のように記述すると、`kage` がタスク開始時（LLM 呼び出し前）にシェルコマンドを実行し、その標準出力をプロンプトに直接展開します。
3. **安全な評価**:
   - 変数の展開は `--dry-run` モード（既存提案）でも確認できるようにし、意図しない破壊的コマンドが事前評価されないよう、基本的には「読み取り専用」のコンテキスト収集に用いることを推奨します。

## Inspiration

- **GitHub Actions**: `${{ github.sha }}` や `${{ env.VAR_NAME }}` による強力なコンテキスト注入。
- **Taskfile / Makefile**: `$(shell git rev-parse HEAD)` のような事前評価。
- **Promptfoo / Prompty**: LLM プロンプトに対する外部データソースや変数の事前バインディング。

## Expected Daily Benefit

エージェントに「情報を探させる」手間を省き、「情報がすでに揃った状態から推論・作業をスタートさせる」ことができます。これにより、夜間バッチの実行速度が向上し、不要なツール呼び出しエラーによる失敗（flaky な動作）が激減します。タスクの意図も静的なプロンプトとしてより明確に記述できるようになります。

## Scope and Non-Goals

- **Scope**: タスク開始直前に Markdown 本文内の特定のプレースホルダーをパースし、組み込み変数や軽量な `exec` の標準出力と置換するプリプロセッサの実装。
- **Non-Goals**: 高度なテンプレートエンジン（条件分岐 `if` やループ `for` など）の導入はスコープ外とします。複雑なロジックが必要な場合は、これまで通りエージェントに自律的にスクリプトを書かせて実行させる設計を維持します。

## Risks or Open Questions

- **`exec` のセキュリティ**: タスクファイルに `{{ exec "rm -rf /" }}` が書かれていた場合、LLM の介入なしに実行されてしまうリスク。`exec` で許容するコマンドを制限するか、フロントマターで `allow_exec_variables: true` のような明示的な許可を求めるべきか。
- **エスケープ処理**: 意図せず `{{ }}` という文字列をプロンプトに含めたい場合のエスケープ構文（例: `\{{ }}`）の定義が必要。

## Suggested Rollout

1. **Phase 1**: `{{ date.today }}` や `{{ env.* }}` などの安全で静的な組み込み変数の置換処理をタスクロード時に実装。
2. **Phase 2**: `{{ exec "..." }}` サポートの追加と、実行タイムアウト（数秒で終わらないコマンドはエラーにする）の安全策の導入。
3. **Phase 3**: 先行提案である `--dry-run` と連携し、展開後の最終的なプロンプトをユーザーが事前確認できる体験の提供。
