# タスク実行のタイムゾーンおよび時間帯制限 (timezone, allowed_hours, denied_hours) のサポート

## Summary

`kage` のタスク定義 (Markdown frontmatter) において、`timezone`、`allowed_hours`、および `denied_hours` を指定できるようにし、サーバーのシステムタイムゾーンに依存せず、かつ複雑な cron 式を書かずに「特定の時間帯だけ動かす」「深夜やメンテナンス中は動かさない」といった制御を可能にする提案です。

## Problem

現在、`kage` は `cron` による定期実行に依存しており、実行スケジュールは標準的な cron 式で表現されます。しかし、以下の課題があります。

1. **タイムゾーンの不一致**: サーバーが UTC で動いている場合、JST (日本時間) での `09:00` にタスクを実行したいなら `0 0 * * *` と記述しなければならず、認知負荷が高く直感的ではありません。
2. **実行禁止時間帯の表現の難しさ**: 「毎時実行するが、夜間（02:00〜05:00）は除く」といった設定を cron 式だけで記述するのは非常に複雑で、バグを生みやすいです。
3. **柔軟な一時停止**: 外部 API のメンテナンス時間や、利用ユーザーの「就業時間内のみ動作してほしい」という要望に素早く対応するためのわかりやすい設定項目がありません。

## Proposal

タスク定義ファイル（例: `.kage/tasks/daily_task.md`）の frontmatter に以下のフィールドを追加します。

```yaml
---
name: "daily_task"
cron: "0 * * * *" # 毎時0分に実行
timezone: "Asia/Tokyo"
allowed_hours: ["09:00-18:00"]
denied_hours: ["12:00-13:00"] # お昼休みは避ける
---
```

1. **`timezone` の解釈**: `cron` で指定したスケジュールと、`allowed_hours` / `denied_hours` で指定した時間をこのタイムゾーン基準で評価します。
2. **`allowed_hours` の評価**: この時間帯（例: 09:00〜18:00）に該当しない場合、cron がトリガーされても `kage` はタスクを実行せず即座に終了（スキップ）します。
3. **`denied_hours` の評価**: `allowed_hours` 内であっても、この時間帯に含まれている場合はタスクをスキップします。

## Inspiration

- **GitHub Actions**: cron トリガーは UTC 固定であるため、JST ユーザーから不満が出やすく、それを回避するための専用アクションなどが存在します。`kage` はネイティブで `timezone` をサポートすることでこの課題を解決します。
- **Airflow / Dagster**: タイムゾーンアウェアなスケジューリングと、SLA や実行可能ウィンドウの概念を持ち、安全な運用を実現しています。
- **iOS / Android の通知設定**: 「おやすみモード (Quiet Hours)」の概念。エージェントにも「寝る時間」を教えることで、夜間の予期せぬ API 課金や不要なエラー通知を防ぎます。

## Expected Daily Benefit

- **タスク作成の簡略化**: 毎時実行 (`0 * * * *`) と `allowed_hours` を組み合わせるだけで、日中だけの定期実行を簡単に書けるようになります。
- **運用の安全性向上**: 深夜の API メンテナンス時間帯を `denied_hours` に指定するだけで、エラーによる無駄なリトライや通知のノイズを減らせます。
- **ポータビリティ**: 開発環境（ローカル Mac, JST）と本番環境（Linux VPS, UTC）でシステムタイムゾーンが違っても、タスク定義を修正せずにそのまま移行できます。

## Scope and Non-Goals

- **スコープ内**:
  - `timezone` (tz database name), `allowed_hours`, `denied_hours` をパースし、実行可否を判定するロジックの追加。
  - スキップされた場合の `.kage/memory/` への軽量なスキップログの記録（毎時記録され続けるのを防ぐため、連続スキップ時のログ圧縮も検討）。
- **スコープ外**:
  - `cron` 以外の複雑なカレンダーベースのスケジュール（祝日対応など）は当面サポートしません。
  - 過去のスキップされたタスクを後から強制的に一括実行するバックフィル機能は含めません。

## Risks or Open Questions

- **スキップログの肥大化**: 毎分実行タスクが `denied_hours` に引っかかり続けると、スキップログが大量に生成される可能性があります。連続スキップ時は最初の1回だけログを残すか、UI 側でまとめて表示する工夫が必要です。
- **cron デーモンとの責務分界点**: `kage` 内部でスキップするため、OS の cron デーモン自体は起動し、`kage` プロセスが走るオーバーヘッドは残ります。軽量なプロセス立ち上がりと終了になるよう実装を最適化する必要があります。

## Suggested Rollout

1. `pydantic` モデルなどのタスク定義スキーマに `timezone`, `allowed_hours`, `denied_hours` (共にオプショナル) を追加。
2. `pytz` または `zoneinfo` を用いて、現在時刻を指定タイムゾーンで評価するユーティリティ関数を実装。
3. タスク実行の最序盤（エージェントや LLM クライアントの初期化前）に評価ロジックを挟み、条件外なら Exit Code 0 (または専用の Skip Code) で終了する挙動を追加。
