# タスク実行時の LLM Provider 自動フォールバック (Provider Fallback)

## Summary

タスクのフロントマターに `fallback_providers: ["gemini", "openai"]` などのリストを指定できるようにし、メインの AI API が夜間にダウンしたりレートリミットに達した場合に、自動的に別の Provider に切り替えて実行を継続する機能の提案。

## Problem

`kage` の最大の提供価値は「Go to sleep. Wake up to results. (寝ている間に結果を出す)」ですが、AI プロバイダーの API (Claude, OpenAI, Gemini など) は、深夜帯のメンテナンスや突発的な障害、あるいはレートリミット (HTTP 429) によってリクエストが失敗することが珍しくありません。
現在の仕様では、単一の `provider` に依存しているため、API エラーが発生するとタスクはそのまま異常終了してしまいます。結果として、ユーザーは翌朝「完了した成果物」ではなく「API の 5xx エラーログ」を見ることになり、自律実行の信頼性が大きく損なわれています。

## Proposal

タスク定義 (`.kage/tasks/*.md`) のフロントマターに `fallback_providers` フィールドを追加し、API 障害時のプロバイダールーティングをサポートします。

```yaml
---
name: Overnight Batch Processor
cron: "0 2 * * *"
provider: claude
fallback_providers: ["gemini", "openai"]
mode: autostop
---
```

1. **自動フォールバック**: メインの `provider` が HTTP 5xx エラー、ネットワークタイムアウト、または 429 Too Many Requests で失敗した場合、設定された順序で次のプロバイダー (例: `gemini`) を使用してタスク（または失敗したサブタスク）を再試行します。
2. **ログと通知の可視化**: ダッシュボードや Slack/Discord 通知において、「Claude がダウンしたため Gemini にフォールバックして完了しました」というメタデータを明記し、ユーザーが事後に状況を把握できるようにします。
3. **グローバル設定でのデフォルトフォールバック**: `~/.kage/config.toml` にデフォルトのフォールバックチェーンを定義できるようにし、各タスクファイルへの記述を省略可能にします。

## Inspiration

- LiteLLM や Vercel AI SDK などの LLM ルーティング / フォールバック機能。
- CI/CD パイプラインにおけるリトライやフォールバックジョブの概念。

## Expected Daily Benefit

夜間のバッチタスクや重いコード監査タスクが、単一の AI プロバイダーの一時的な障害によって頓挫するのを防ぎます。これにより、「朝起きれば必ず何らかの成果物が出ている」という `kage` のコア体験の信頼性が飛躍的に向上し、日々の運用における心理的安心感（DX）が高まります。

## Scope and Non-Goals

- **スコープ内**: API 通信エラー、5xx サーバーエラー、429 レートリミット時のプロバイダー切り替え。
- **非目標 (Non-Goals)**: 出力品質が悪い（ハルシネーション等）ことを理由としたフォールバック。あくまでシステム・ネットワークレベルのエラーに対するフェイルセーフとして機能させます。
- **非目標**: プロバイダーごとのコスト最適化（安い順に試す等）の自動化。静的なリストの順序に従うのみとします。

## Risks or Open Questions

- **コンテキストウィンドウの差異**: `claude` と `gemini` で扱える最大トークン数や画像入力のサポート状況が異なるため、フォールバック先でコンテキスト溢れや機能非互換エラーが起きる可能性があります。その場合はさらに次のフォールバック先を試すか、Graceful に失敗させる必要があります。
- **システムプロンプトの互換性**: 各モデルの特性（フォーマット追従性など）が異なるため、特定のモデルに最適化されたタスク指示がフォールバック先で意図通りに動かないリスクがあります。

## Suggested Rollout

1. まずは設定ファイルのスキーマ (`pydantic` モデル等) に `fallback_providers` を追加。
2. コアの実行エンジン (CLI ラッパー部分) で、API 呼び出しの例外をキャッチする機構を実装し、リストに従って順次再試行するロジックを追加。
3. ダッシュボードおよび通知メッセージにフォールバック発生のフラグを追加し、試験運用を開始。
