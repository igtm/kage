# Conditional Task Execution based on File Changes

## Summary
`cron` スケジュールで定期実行されるタスクにおいて、「特定のファイルやディレクトリに変更があった場合のみ」AIプロバイダやコマンドを実行する `condition.changed_files` オプションを導入します。

## Problem
現在、`kage` のタスクは `cron` ベースで実行されますが、毎時や毎10分といった高頻度で実行されるタスクの場合、対象となるコードやデータに全く変更がないにも関わらずプロバイダAPIを呼び出してしまい、API予算（トークン）を無駄に消費したり、不要なログが蓄積したりする問題があります。
また、ユーザーは「前回の実行からソースコードが更新された時だけレビューしてほしい」「仕様書が書き換わった時だけPoCをアップデートしてほしい」といった「変更駆動」のタスクを定義したいことがよくあります。

## Proposal
タスク定義のフロントマター（YAML）に `condition` セクションを追加し、実行条件を宣言的に記述できるようにします。

```yaml
---
name: Auto-Review
cron: "*/15 * * * *"
provider: claude
condition:
  changed_files:
    - "src/**/*.ts"
    - "docs/**/*.md"
---
```

`kage` の実行エンジンは以下のステップを踏みます：
1. `cron` によってタスクがキックされる。
2. 前回タスク実行完了時のGitのコミットハッシュやファイルの更新日時（タイムスタンプ）を `kage.db` または `.kage/memory/` から取得。
3. `condition.changed_files` の glob パターンに一致するファイルが、前回実行時から変更されているかを判定する（`git diff` またはファイルの `mtime` 比較）。
4. 変更がなければ、AI呼び出しをスキップしてタスクを終了（Skipped状態としてログにのみ記録）。変更があれば通常通り実行。

## Inspiration
- **GitHub Actions (`paths` / `paths-ignore`)**: ワークフローのトリガー条件として、特定パスのファイル変更を指定できる機能。
- **`make` / `just`**: ターゲットの依存ファイルが更新されたときのみビルドを実行する仕組み。

## Expected Daily Benefit
- 無駄なAI API呼び出しを劇的に削減し、コストとAPIレートリミット超過を防ぎます。
- ユーザーは「変更がなければ何も起きない」という安心感のもと、より高頻度（例: 5分おき）でエージェントを回すことができるようになり、フィードバックループが高速化します。

## Scope and Non-Goals
- **Scope**: ファイルパスの glob マッチングによる変更検知、およびGit追跡下/非追跡下の両方のサポート（GitリポジトリであればGitの差分を活用し、そうでなければmtimeベース等）。
- **Non-Goals**: リアルタイムなファイルシステムイベント（inotifyやfsevents）の監視機能（daemonlessという思想に反するため、あくまでcronベースの定期チェックによるポーリング型とします）。

## Risks or Open Questions
- ファイル変更の基準を「前回のタスク成功時」にするか「前回のタスク実行時（失敗含む）」にするか。成功時を基準にする方が安全（エラーで止まった場合に再試行されるため）。
- 新規ファイル作成やファイル削除に対する検知漏れを防ぐため、単純なタイムスタンプだけでなくディレクトリの状態も見る必要がある。

## Suggested Rollout
1. まずは `kage.db` またはタスクメモリに「最後に処理したコミットハッシュ（またはタイムスタンプ状態）」を保存する仕組みを追加。
2. タスク定義のパーサーを拡張し `condition.changed_files` を読み込めるようにする。
3. `kage run` の実行パイプラインの初期フェーズに、条件判定（Condition Check）ステップを挿入し、条件を満たさない場合は早期リターンする。
