# アラート疲労を防ぐ連続エラー通知の自動ミュート (Alert Fatigue Protection)

## Summary

同一タスクが連続して失敗し続けた際、Slack や Discord へのエラー通知頻度を自動的に減衰（Exponential Backoff）させるか、一定回数で一時停止（Auto-Snooze）することで、ユーザーのアラート疲労（Alert Fatigue）を防ぐ機能の提案。

## Problem

`kage` は cron ベースで動作するため、`continuous` モードで数分〜数十分おきに実行される監視タスクやバッチタスクが設定されることがよくあります。
しかし、対象の外部 API が長時間ダウンしていたり、プロンプトの記述ミスで LLM が毎回パースエラーを起こしたりすると、cron が発火するたびに毎回同じエラー通知が Connector 経由で送信されてしまいます。
朝起きたとき、チャットツールに「数百件の同じエラー通知」が溜まっていると、他の重要な通知（本当に見てほしい成功結果など）が埋もれてしまい、結果的にユーザーが通知自体を無視するようになる「オオカミ少年状態（アラート疲労）」に陥る課題があります。

## Proposal

タスクのフロントマター、または `.kage/config.toml` のグローバル設定に `alert_policy`（通知抑制ルール）を導入します。

```yaml
---
name: "Frequent API Monitor"
cron: "*/5 * * * *"
mode: continuous
alert_policy:
  mute_after_failures: 3
  backoff: true
---
```

1. **連続失敗のトラッキング**: `kage.db` またはタスクのステートとして「連続失敗回数 (consecutive failures)」を記録します。
2. **通知の指数的減衰 (Exponential Backoff)**: `backoff: true` の場合、失敗のたびに毎回通知するのではなく、1回目、2回目、4回目、8回目の失敗時にのみエラー通知を送信します。
3. **自動ミュート (Auto-Snooze)**: 失敗が `mute_after_failures` の上限（例: 3回）に達した場合、「同一エラーが続いているため、このタスクの通知をミュートしました」という最終通知を送り、以降はタスクが成功して回復（Recover）するまでエラー通知を送信しません（実行自体はスケジュール通り行われ、内部ログには記録されます）。
4. **回復通知 (Recovery Alert)**: ミュート期間中にタスクが成功した場合、連続失敗回数をリセットし、「タスクが正常な状態に回復しました」という通知を1回だけ送ります。

## Inspiration

- **Datadog / PagerDuty**: アラートのグルーピングや、同一インシデントの連続通知を抑制するインシデント管理プラットフォームの標準機能。
- **GitHub Actions**: スケジュール実行が連続で失敗した場合にのみ通知を送り、回復した時にもう一度送る仕組み。

## Expected Daily Benefit

エラーの洪水によってチャットツールが破壊されるのを防ぎ、心理的な負担を大きく軽減します。
別の提案である「Morning Briefing (朝のダイジェスト通知)」は成功やサマリーの認知負荷を下げるものですが、本提案は「リアルタイムな緊急エラー通知」のノイズを減らすことに特化しており、「真に必要な介入要請」だけがユーザーに届くようになります。

## Scope and Non-Goals

- **Scope**: 同一タスクの連続失敗回数のカウントと、Connector への送信直前でのフィルタリング（インターセプト）処理。回復時の Recovery 通知。
- **Non-Goals**: エラーの内容（スタックトレースの中身）まで解析して「新しい種類のエラーなら通知する」といった高度な重複排除（Deduplication）は初期スコープ外とし、シンプルに「タスクレベルでの連続失敗」のみを判定基準とします。

## Risks or Open Questions

- **状態管理の複雑化**: 通知を送るかどうかの状態を `kage.db` に持たせる必要があります。プロセスがまたがるため、DB のロックやトランザクションを正しく扱う必要があります。
- **一時停止 (Pause) との違い**: このミュート機能は「通知」のみを止めるものであり、タスクの「実行」自体は止まりません。もし実行自体も止めたい場合は、別途エラー時に `status: paused` へ自動移行するサーキットブレーカー（Circuit Breaker）的なアプローチと整理・統合する方が美しいかもしれません。

## Suggested Rollout

1. `kage.db` の該当タスクレコードに `consecutive_failures` カラム（または JSON 内のフィールド）を追加。
2. タスク実行完了（エラー終了）時に、このカウントをインクリメント。成功時は `0` にリセット。
3. Connector 送信モジュール内で、`consecutive_failures` とタスクの `alert_policy` を比較し、送信をスキップするロジックを実装。
4. （将来）閾値を超えた場合に通知ミュートではなくタスク自体を `paused` にする Circuit Breaker 設定のサポート。
