# タスク定義の共通化と再利用 (Task Templates / Blueprints)

## Summary

頻繁に作成する類似タスク（例: コードレビュー、ログ解析、定期リファクタリング、翻訳）のボイラープレートを排除するため、ベースとなる構成を `.kage/templates/` に定義し、個別のタスクファイルからは `template:` 参照と固有の変数 (`vars:`) を指定するだけでエージェントをデプロイできる機能の提案。

## Problem

`kage` は `.kage/tasks/*.md` にマークダウンを配置するだけで自律エージェントを稼働させることができますが、タスクの数が増えてくると以下のような DX の課題が生じます。

1. **冗長なボイラープレート**: タスクごとにフロントマター（`provider`, `mode`, `timezone` など）や、定型プロンプト（「あなたはシニアエンジニアです」「必ず日本語で出力し、ファイルに保存してください」等）を毎回コピー＆ペーストする必要があります。
2. **メンテナンスのトイル**: 例えば「使用するモデルを `gemini-2.0-flash` から `gemini-2.5-pro` に変更したい」「ベースとなるプロンプトの指示を改善したい」と思った際、数十個のタスクファイルを一斉に書き換える手間が発生します。
3. **共有の難しさ**: プロジェクト内で「良いプロンプト」や「安定した実行設定」が見つかっても、それを他タスクへ再利用する標準的な仕組みがありません。

## Proposal

テンプレートエンジンを導入し、ベースとインスタンスを分離します。

1. **テンプレートの定義 (`.kage/templates/`)**:
   プロジェクト（またはグローバル）にテンプレートディレクトリを設け、共通の振る舞いを記述します。
   例: `.kage/templates/code-review.md`
   ```yaml
   ---
   provider: "gemini"
   mode: "continuous"
   ---
   あなたはシニアエンジニアです。以下のディレクトリを対象にコードレビューを行い、結果を Markdown で保存してください。
   対象: {{ vars.target_dir }}
   特に {{ vars.focus_area }} の観点を重視してください。
   ```

2. **個別タスクからの参照**:
   実際のタスクファイル (`.kage/tasks/frontend-review.md`) では、テンプレート名と固有の変数のみを記述します。
   ```yaml
   ---
   template: "code-review"
   cron: "0 2 * * *"
   vars:
     target_dir: "src/frontend"
     focus_area: "React hooks の依存配列とパフォーマンス"
   ---
   ```

3. **実行時のマージ解決**:
   `kage` は実行時にテンプレートのフロントマターとタスクのフロントマターをマージし（タスク側を優先）、プロンプト内のプレースホルダを `vars` の値で置換してから AI に渡します。

## Inspiration

- **GitHub Actions**: `uses:` キーワードによる Reusable Workflows (`workflow_call`) や Custom Actions の再利用。
- **Helm / Kustomize**: ベースとなるマニフェスト（Base/Template）に対して、環境固有の差分（Overlays/Values）を注入・上書きする宣言的アプローチ。
- **Ansible Roles**: 共通のタスク定義を Role として切り出し、変数を与えて再利用する仕組み。

## Expected Daily Benefit

新しいタスクをスケジュールする際の心理的ハードルが劇的に下がり、「数行の YAML を書くだけで強力な定型エージェントが動き出す」体験が得られます。
また、プロジェクト全体でプロンプトエンジニアリングのベストプラクティスを一元管理できるようになり、LLM の出力品質の底上げや一括アップデートが容易になります。日々の運用における認知負荷と設定ファイルの変更コストが大幅に削減されます。

## Scope and Non-Goals

- **Scope**: ローカル `.kage/templates/` の読み込み、フロントマターの階層的なマージ（Shallow Merge 程度）、および `vars` を用いたプロンプト内の変数展開機能の実装。
- **Non-Goals**: 高度な制御構文（for ループや複雑な if 分岐）を持つ巨大なテンプレートエンジンの組み込みは、`kage` の「薄く透明なレイヤー」という思想に反するため見送ります。シンプルなプレースホルダのテキスト置換（Mustache/Jinja2 サブセット）に留めます。

## Risks or Open Questions

- **マージロジックの複雑化**: テンプレートが別のテンプレートを継承する（多重継承）ことを許容すると、最終的なプロンプトの予測・デバッグが困難になります。最初は「1階層（Base -> Task）のみ」に制限するのが安全です。
- **既存の変数構文との衝突**: 別の提案である「Dynamic Context Variables（`{{ date.today }}` 等の動的変数）」の記法とパーサーレベルで競合しないよう、統一された構文と評価タイミングの設計が必要です。

## Suggested Rollout

1. **Phase 1**: `.kage/templates/*.md` からのベースフロントマターの読み込みと、タスク側でのマージ機能（変数の置換なし）の実装。
2. **Phase 2**: タスクのフロントマターに `vars:` を追加し、本文中のプレースホルダ（`{{ vars.xxx }}`）を静的にテキスト置換する機能の追加。
3. **Phase 3**: グローバルな設定ディレクトリ（例: `~/.kage/templates/`）からのテンプレートフォールバック読み込みサポート。
