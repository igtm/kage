# タスク完了時の自動 Pull Request 作成機能 (Auto PR Creation)

## Summary
タスクが自律的にコードを変更して完了した際、自動的にブランチを作成・コミットし、メインブランチに対する Draft Pull Request を作成する機能の提案。

## Problem
`kage` の魅力は「夜間にタスクを放置し、朝に成果を得る」ことですが、現在の仕様では成果物（変更されたコード）が作業ディレクトリに直接書き込まれ、dirty な状態のまま放置されます。
翌朝ユーザーが PC を開いたとき、手動で `git diff` を確認し、適切な粒度でコミットメッセージを書き、ブランチを切って PR を作成する手間（トイル）が発生します。
また、複数のタスクが同じ夜間に並行して動作した場合、どの変更がどのタスクによるものか混ざってしまい、切り分けが困難になるリスクがあります。

## Proposal
タスクのフロントマター（`.kage/tasks/*.md`）に `post_action: create_pr` または `auto_pr: true` を設定できるようにします。
これが有効なタスクが `mode: autostop` や単発実行で「完了 (Completed)」ステータスになった際、`kage` 実行レイヤーが以下の操作を自動で行います。

1. **ブランチ作成**: `kage/{task_slug}-{YYYYMMDD}` のような固有のブランチを現在の作業ツリーから切り出します。
2. **コミット**: タスクの実行内容や生成されたメモリのサマリーをもとに、AI を使って適切なコミットメッセージを生成し、変更をコミットします。
3. **Push & PR**: リモートにプッシュし、タスクのログ（`kage.db` や `.kage/memory/` の該当部分）を PR の本文に記載した状態で Draft PR を作成します。
4. **ワークツリーの復帰**: 作業ディレクトリを元の状態（`main` 等）に戻し、次のタスクがクリーンな状態で走れるようにします（設定でトグル可能）。

```yaml
---
name: Overnight Refactoring
cron: "0 2 * * *"
mode: autostop
auto_pr: true
---
```

## Inspiration
- **Claude Code / Aider**: エージェントが自律的にコミットを行う機能。
- **Renovate / Dependabot**: 依存関係の更新を PR として人間にお膳立てする UX。人間は朝起きて「Approve & Merge」を押すだけという体験。

## Expected Daily Benefit
- 朝起きたときの認知負荷が激減します。「何が変更されたか調べる」フェーズをスキップし、すぐに GitHub (またはエディタのPR拡張) 上での「レビューとマージ」の意思決定に集中できます。
- 複数タスクがそれぞれ独立した PR として提案されるため、変更が混ざる事故を防ぎ、個別に承認・リジェクトが可能になります。

## Scope and Non-Goals
- **Scope**: タスク完了時の git 操作、PR 作成 API 呼び出し、および元のブランチへの復帰処理。
- **Non-Goals**: 複雑な Git コンフリクトの自動解消。マージ時にコンフリクトが起きた場合は、人間が手動で解決するか、PR 内で再度エージェントに解決を指示する前提とします。

## Risks or Open Questions
- **未完了時の変更**: `autostop` モードで途中のサブタスクで止まった（あるいはタイムアウトした）場合、PR は作成せずにそのままにするか、WIP の PR を作るか。
- **認証**: `gh` CLI の呼び出しに依存するか、`kage` 自身に GitHub トークンを持たせるか。ローカル環境であれば既存の `gh` や SSH 鍵を使うのが最もシームレス。
- **複数回の修正**: 人間が PR にレビューコメントを書いた後、次の夜間にエージェントがその PR のブランチ上でタスクを再開（Resume）して追加コミットできるとさらに強力。

## Suggested Rollout
1. まずは `kage run --create-pr` のような CLI オプションとして手動タスク実行時の PR 作成機能を実装する。
2. その後、タスクのフロントマター設定による自動トリガーと、Slack/Discord コネクターへの「PR 作成完了」通知を統合する。
