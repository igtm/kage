# Task-Level Resource Mutex Locks

## Summary
タスクの Frontmatter に `locks` フィールドを追加し、複数のタスクが同時に同じファイルやポート、あるいは抽象的なリソースを操作して競合することを防ぐ、タスク間の排他制御（Mutex）メカニズムを導入する提案です。

## Problem
`kage` は cron や launchd ベースで複数のタスクを並行して実行できます。しかし、複数の自律型 AI タスクがたまたま同じ時間帯に実行され、同じファイル群（例: `src/api/` のルーティング定義）を編集しようとしたり、テスト実行用に同じローカルポート（例: `8080`）をバインドしようとしたりすると、不整合やクラッシュが発生します。
現在、タスクの多重起動制御（`allow`, `forbid`, `replace`）は「同一タスク内」の制御に留まっており、「異なるタスク間」のリソース競合を防ぐ仕組みがありません。結果として、実行タイミングが被らないように人間が `allowed_hours` で時間をずらすなどの泥臭い管理を強いられています。

## Proposal
タスクの Frontmatter に `locks` フィールドを新設し、実行前に SQLite データベース（`kage.db`）を用いてリソースロックを取得する仕組みを実装します。

```markdown
---
mode: continuous
locks:
  - "file:src/api/*"
  - "port:8080"
  - "custom:db-migration"
lock_timeout: "5m"
---
# APIの自動テストとリファクタリング
...
```

- ロックの種類は `file:`、`port:`、`custom:` などをサポート。
- タスク開始時に必要なロックの取得を試み、他のタスクが既にロックを保持している場合は、その回の実行をスキップ（または `lock_timeout` まで待機）します。
- タスク終了時、またはクラッシュ時に確実にロックを解放します（データベース上のセッション/PID監視による自動解放を含む）。

## Inspiration
- **GitHub Actions (Concurrency Group)**: 同じ concurrency group 内のジョブをキャンセル・待機させる仕組み。
- **Airflow / Dagster (Resource Pools)**: 限られたリソース（DBコネクションなど）に対する同時実行数を制御する仕組み。
- **Linux `flock`**: OS レベルのファイル排他制御。

## Expected Daily Benefit
- **衝突の不安払拭**: 関連するタスク群を気にせず `continuous` で回しておけるようになり、並列処理時の Git 競合や環境破壊を未然に防ぎます。
- **運用コストの低下**: タスクごとに緻密なスケジュール（何時何分に実行するか）を人間がパズルする必要がなくなり、単に「必要なリソース」を宣言するだけで安全にオーケストレーションされます。
- **デバッグの手間削減**: 原因不明のテスト失敗やファイル破壊が、実は「他のAIタスクの裏側での介入」だったというケースを撲滅できます。

## Scope and Non-Goals
- **Scope**: 同一 PC（同一 SQLite DB）上でのローカルなタスク間排他制御の実装。
- **Non-Goals**: 分散環境（複数マシン間）での分散ロック（Redis などの外部依存）の実装はスコープ外とします。あくまでスタンドアロンな `kage` の範囲に留めます。また、デッドロックの高度な検知・解決アルゴリズムも当面は実装せず、シンプルなタイムアウトベースで対処します。

## Risks or Open Questions
- **ロックの残留**: kage プロセスが `kill -9` などで強制終了された場合、ロックが DB に残ったままになるリスクがあります。OS の PID チェックなどを併用して、死んだプロセスのロックを自動回収（Stale Lock 判定）する堅牢な実装が必要です。
- **待機かスキップか**: ロックが取れなかった場合、「取れるまで待つ」のか「今回は諦めて次の cron までスキップする」のか。デフォルトはスキップの方が cron ベースの思想（ステートレス）に合致すると考えられます。

## Suggested Rollout
1. `kage.db` に `resource_locks` テーブルを追加。
2. タスクパース時に `locks` フィールドを読み取る処理を追加。
3. 実行エンジンの開始直後と終了直前に、ロックの取得・解放ロジックを挿入。
4. Stale Lock（所有プロセスが存在しないロック）の自動クリーンアップ機構を実装。
