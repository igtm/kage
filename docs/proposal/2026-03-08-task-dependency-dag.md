# タスク間の依存関係 (DAG) サポート (`depends_on`) の導入

## Summary

タスクのフロントマターに `depends_on: ["task_a"]` を指定できるようにし、先行タスクが正常終了していない場合は自身の実行をスキップ（または保留）する依存関係（DAG）機能の提案。

## Problem

現在の `kage` は各タスクが OS の `cron` や `launchd` に独立して登録・実行される仕組みです。そのため、「Task A（データ収集）が終わってから、Task B（AI分析）を実行する」といった順序関係を表現するには、cron の実行時間を「0:00」と「1:00」のようにずらす運用でカバーするしかありません。
しかし、Task A がエラーで失敗した場合でも Task B が時間通りに起動してしまい、無効なデータで AI API を叩いて無駄なコストを消費したり、後続処理が予期せぬエラーを起こすという問題（連鎖障害）があります。

## Proposal

タスクのフロントマターに `depends_on` フィールドを追加し、シンプルなタスク間依存を記述できるようにします。

1. **フロントマターでの依存定義**:
   ```yaml
   name: AI Analysis
   cron: "0 1 * * *"
   depends_on:
     - "data_fetch_task"
   ```
2. **実行時の依存チェック**:
   `kage` が OS スケジューラから起動された際、ロジック内で `kage.db` または `.kage/memory/` を参照し、指定された先行タスク（`data_fetch_task`）が「本日（または一定時間内）に正常終了 (`status: completed`) しているか」をチェックします。
3. **未完了時の挙動**:
   条件を満たさない場合、Task B は `skipped (waiting for data_fetch_task)` として直ちに終了し、AI を呼び出しません。

## Inspiration

- **Airflow / Dagster / make**: 依存するジョブやターゲットが完了してから後続を動かす仕組み。
- **GitHub Actions**: `needs: [job1]` によるジョブ間の順序制御。

## Expected Daily Benefit

夜間バッチにおいて、「前処理タスクがコケたのに後続の重いAIタスクが空振りで走ってAPI料金を浪費する」といった事故を完全に防げます。時間ベースの不安定なスケジューリングから、状態ベースの堅牢な実行フロー（Fail-fast）へと移行でき、夜間自動化の信頼性が大幅に向上します。

## Scope and Non-Goals

- **Scope**: 同一プロジェクト内のタスク間における「状態チェック」による簡易的な DAG（有向非巡回グラフ）のサポート。
- **Non-Goals**: OS のスケジューラを置き換えるような、kage 自身が常駐してタスクを連鎖起動する仕組みは実装しません（kage は常駐しない思想のため）。あくまで cron で起動されたタイミングでの「実行可否のガード条件」として機能させます。

## Risks or Open Questions

- **状態のリセットタイミング**: 毎日実行するタスクの場合、「先行タスクが完了した」という状態をいつリセットするかの定義が必要です（例: 日付が変わったらリセット、または実行の有効期限 `valid_for: 24h` などを設けるか）。
- **デッドロックの防止**: `depends_on` が循環参照（A -> B -> A）になっていないかを `kage doctor` やタスク保存時に静的解析して警告する仕組みが必要です。

## Suggested Rollout

1. **Phase 1**: `kage.db` のタスク実行履歴テーブルの拡張と、フロントマターの `depends_on` パース処理の実装。
2. **Phase 2**: タスク実行開始時の前提条件バリデーション処理（依存未解決時の skip ログ出力）の追加。
3. **Phase 3**: 循環参照を検知する `kage doctor` の強化と、kage UI 上での依存グラフ（ノードツリー）の可視化。
