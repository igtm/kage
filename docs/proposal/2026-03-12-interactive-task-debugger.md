# Interactive Task Debugger (`kage debug`)

## Summary
夜間やバックグラウンドで失敗したタスクに対して、その失敗直後のコンテキスト（メモリ、直近のエラーログ、中途半端なファイル状態）を保持したまま、CLI上で対話的にエージェントと問題解決を行う `kage debug <task_name>` コマンドの追加提案。

## Problem
`kage` の強みは OS ネイティブなスケジューラによる「完全な自律・バックグラウンド実行」ですが、タスクが途中でエラー終了したり、LLMのハルシネーションで意図しない方向に進んでしまった場合、リカバリの認知負荷が高くなります。
現在はユーザーがエラーログを読み、タスクのMarkdown指示を修正し、次の cron 実行（または `kage run`）を待つ必要があります。失敗した「その場」の文脈をエージェントと共有しながら「どこで躓いたのか？」「このライブラリを使ってやり直してみて」と直接軌道修正する手軽な手段が CLI に欠けています。

## Proposal
失敗または `paused` 状態のタスクに対して、専用の対話モードを起動するコマンドを提供します。

```bash
$ kage debug ocr_benchmark
> Task 'ocr_benchmark' failed at sub-task "Benchmark EasyOCR" with error: "CUDA out of memory".
> Entering debug mode. You are now chatting with the agent in the exact context of the failure.

User: GPUメモリが足りないみたいだから、バッチサイズを下げて再実行してみて。
Agent: わかりました。`test_easyocr.py` のバッチサイズを 1 に修正し、該当のサブタスクから再開します。
...
```

**内部動作**:
1. `kage debug` 実行時、指定タスクの最新の `.kage/memory/` ファイルと直近のエラーログ（標準エラー出力）をシステムプロンプトに動的に注入する。
2. 標準の CLI チャット REPL を起動し、ユーザーとエージェント間で会話を行う。
3. エージェント側で解決策が合意できたら、ユーザーの指示でそのまま修正作業を行わせる。

## Inspiration
- **Aider / Claude Code**: ターミナルでの対話型エージェント開発体験。
- **Python の `pdb` / `breakpoint()`**: プログラムがクラッシュした「その状態」にフックして調査・修正するデバッガの概念を、AIタスクに持ち込む。

## Expected Daily Benefit
- **圧倒的なトラブルシューティングの高速化**: 失敗したタスクの再開や修正にかかる時間が劇的に短縮されます。エラーログをコピーして別の Web ブラウザの AI に貼り付ける手間がゼロになります。
- **タスク成功率の向上**: 一度失敗しても、翌朝数分の対話的な「軌道修正（ヒント出し）」を行うだけで、コンテキストを失わずにタスクを完了に導けるようになります。

## Scope and Non-Goals
- **Scope**: CLI 上での単一タスクの対話的デバッグ REPL の提供。コンテキスト（直近のメモリ、エラーログ、タスク定義）の自動注入。
- **Non-Goals**: Web Dashboard 側でのフル機能のグラフィカルなデバッガ実装（まずは CLI の DX を優先）。

## Risks or Open Questions
- デバッグモード中のエージェントによるファイル変更と、バックグラウンドの cron 実行がタイミング悪く衝突しないか？（デバッグ開始時にタスクのロックを取得し、cron による発火をスキップする排他制御が必要）。
- デバッグ用に起動する LLM は、元のタスクと同じモデルを使うべきか、ユーザーの操作用により高速/高機能なモデルを割り当てるべきか。

## Suggested Rollout
1. CLI に `kage debug <task_name>` サブコマンドを追加し、標準の対話プロンプトとして起動できるようにする。
2. 起動時に `.kage/memory/` の最新内容とタスクの実行ログを読み込み、プロンプトの冒頭にコンテキストとして注入するロジックを実装。
3. デバッグセッション中の排他制御（該当タスクの cron 実行スキップ）を追加。
