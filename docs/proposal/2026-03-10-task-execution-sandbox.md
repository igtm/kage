# タスク実行の Docker サンドボックス分離 (Task Execution Sandbox)

## Summary

タスクのフロントマターで `sandbox: docker` を指定できるようにし、エージェントが生成・実行するシェルコマンドやスクリプトを一時的な Docker コンテナ内で隔離実行する機能の提案。

## Problem

`kage` はローカルマシンや VPS 上で自律稼働し、必要に応じてシェルコマンドやコードを生成・実行します。しかし、「寝ている間に結果を出す」という運用において、エージェントのハルシネーションやロジックのバグにより、ホスト OS の重要なファイルを誤って削除・上書きしてしまったり、グローバルな環境（Python や Node.js のパッケージなど）を破壊してしまうリスクが常に付きまといます。
現在はエージェントの「常識」やプロンプト上の注意書きに依存していますが、完全に自律化されたシステムにおいて、ホスト上で任意のコードを直接実行させることはセキュリティおよび安全性の観点で大きな心理的ハードル（ブロッカー）となっています。

## Proposal

タスク定義 (`.kage/tasks/*.md`) のフロントマターに `sandbox` の設定を追加し、実行環境をコンテナレベルで隔離できるようにします。

```yaml
---
name: "Untrusted Code Evaluation"
cron: "0 3 * * *"
sandbox:
  type: docker
  image: "python:3.12-slim"
  mounts:
    - "./workspace:/workspace"
---
```

1. **エフェメラルなコンテナの起動**: タスクの実行開始時に、指定されたイメージ（デフォルトは軽量な Ubuntu や Alpine）をベースにした使い捨てのコンテナを起動します。
2. **コマンド実行のプロキシ**: エージェントが要求するシェルコマンドやスクリプト実行（ツールの呼び出し）を、すべて `docker exec` または Docker SDK 経由でこのコンテナ内にルーティングします。
3. **安全なファイルアクセス**: 必要なディレクトリ（プロジェクトの特定のフォルダなど）のみをコンテナにマウント（Bind Mount）し、それ以外のホストファイルシステムへのアクセスを物理的に遮断します。タスク終了後、コンテナは自動的に破棄されます。

## Inspiration

- **OpenHands (旧 OpenDevin) / SWE-agent**: エージェントの行動をすべて Docker コンテナ内に閉じ込めることで、安全な自律的コーディングを実現するアプローチ。
- **GitHub Actions / GitLab CI**: ジョブごとにクリーンな分離環境（Runner）を提供し、副作用をリセットする仕組み。
- **Dev Containers**: プロジェクトごとに隔離された開発環境を定義する概念。

## Expected Daily Benefit

ホスト環境を破壊するリスクが「物理的にゼロ」になります。これにより、例えば「ウェブからスクレイピングした未知のスクリプトをとりあえず実行して解析する」「新しい言語の環境構築からビルドまでを試行錯誤させる」といった、副作用の強いリスクの高いタスクであっても、ユーザーは安心して `kage` に丸投げ（Go to sleep）できるようになります。安全性（Safe-guard）の担保により、日々の自動化のスコープが劇的に広がります。

## Scope and Non-Goals

- **スコープ内**: エージェントが呼び出すコマンド実行レイヤーを Docker コンテナ内にプロキシする仕組みの実装と、マウントパスの制御。
- **非目標 (Non-Goals)**: Docker Compose のような複数コンテナのオーケストレーションや、Firecracker のようなマイクロ VM レベルの強固なマルチテナント分離は当面スコープ外とします。あくまで「個人のローカル環境を守るためのサンドボックス」に特化します。

## Risks or Open Questions

- **パフォーマンスのオーバーヘッド**: コンテナの起動やイメージの Pull に時間がかかると、タスクの実行速度が低下します。既存のイメージを使い回すなどの最適化が必要です。
- **パーミッションの不整合**: コンテナ内で `root` として生成されたファイルが、ホスト側から編集・削除できなくなる問題（UID/GID のマッピング問題）が頻発します。実行ユーザーをホストに合わせるなどのハンドリングが不可欠です。
- **Docker デーモンの依存**: この機能を利用するにはホストに Docker がインストールされ、起動している必要があります。

## Suggested Rollout

1. まずは試験的な機能（Experimental）として、フロントマターの `sandbox` 定義をパースするスキーマを追加。
2. `kage` のツール（Functions）レイヤーにおいて、「ローカルで実行するツール」と「コンテナ内で実行するツール」を透過的に切り替えられるインターフェース（Abstract Executor）を実装。
3. Docker SDK for Python を用いて、タスク開始時にコンテナを起動し、タスク終了時（または異常終了時）に確実に破棄するライフサイクル管理を導入。
