# API コストの安全装置 (Task Execution API Budget Limit) の導入

## Summary

`kage` のタスク実行において、API 利用料金が意図せず高額になるのを防ぐため、タスク単位（およびグローバル全体）での API 予算上限 (`max_budget_usd` など) を設定し、上限に達した場合にタスクを強制停止（Circuit Breaker）する安全装置の提案。

## Problem

`kage` は `continuous` モードや、自律的なリトライループ（自己修復など）を用いて、夜間やバックグラウンドでユーザーの介入なしに動作し続けます。
この「寝ている間に自律動作する」特性上、万が一プロンプトの設計ミス、無駄なループ、あるいは大規模なコンテキストの反復入力が発生した場合、数時間で高額な API 課金（Claude 3.5 Sonnet や GPT-4o などの賢いが高価なモデルを使用している場合）が発生してしまう「API 破産」のリスクが常に伴います。
朝起きたら数万円の請求が確定していた、という事態は、日常的に使う OSS ツールとして最も避けたい DX の最悪ケースです。

## Proposal

タスクのフロントマター、および `.kage/config.toml` に、API コストの上限（Budget Limit）を設定できるようにします。

1. **トークン消費のトラッキング**:
   各 AI プロバイダー（`codex`, `claude`, `gemini` 等）から返却される Input/Output トークン数をトラッキングし、モデルごとの最新の単価テーブルと掛け合わせて推計コスト（USD）を算出します。
2. **設定項目の追加**:
   - フロントマター (タスク単位): `max_budget_usd: 1.50` （このタスクの 1 実行 / または累計での上限）
   - `.kage/config.toml` (グローバル設定): `global_monthly_budget_usd: 50.0` （1ヶ月全体での上限）
3. **強制停止 (Circuit Breaker)**:
   実行前のチェック、または実行ごとのループの合間に現在の推計コストを評価し、上限を超過しそうな場合、あるいは超過した瞬間にタスクを強制的に `aborted` 状態にし、Discord/Slack コネクター経由で「🚨 予算上限到達のため強制停止」という緊急通知を飛ばします。

## Inspiration

- **Aider / OpenHands**: コマンドラインツールとして動かす際に、トークン消費や推計コストを都度表示してユーザーに意識させる仕組み。
- **AWS Budgets / GCP 予算アラート**: クラウドリソースの青天井課金に対する標準的な予防策。
- **AutoGPT**: 連続実行回数の上限設定に加え、費用ベースのストップ機能が要望され実装される流れ。

## Expected Daily Benefit

「寝ている間に勝手に課金が爆発するかもしれない」という心理的障壁（恐怖）を取り除き、安心して重たいリファクタリングや `continuous` な監視タスクを `kage` に任せられるようになります。
日々の運用において「このタスクは 1ドル までなら使っていい」と予算を割り当てることで、コストパフォーマンスを意識したプロンプト改善や、より安価なモデル（Gemini 2.5 Flash など）への切り替え（プロバイダーフォールバック等）を促すきっかけにもなります。

## Scope and Non-Goals

- **Scope**: トークン消費量の記録、推計コストの算出、フロントマターおよびグローバル設定による予算チェックと強制停止（Circuit Break）の実装。
- **Non-Goals**: 厳密なリアルタイムの課金同期（APIプロバイダーの実際の請求額と1セント単位で一致させること）は目指しません。あくまで「トークン数 × 単価テーブル」による推計値での安全装置とします。

## Risks or Open Questions

- 単価テーブルの最新化: 各プロバイダーがモデルの料金を改定した際、`kage` 内部の計算テーブルをどう保守するか（ハードコードするか、外部から定期フェッチするか）。
- プロバイダー側でトークン数が返却されない、あるいはストリーミング応答時のトークン計算のオーバーヘッドをどう扱うか。
- `max_budget_usd` は「1回の cron 実行での上限」なのか「タスクのライフサイクル全体での累計」なのかの明確化（ユースケースによって両方欲しい可能性）。

## Suggested Rollout

まずはフロントマターでの `max_task_budget_usd`（タスク累計での上限）の導入から始め、実行ごとに `.kage/memory/` に累計消費額を記録していく MVP からスタートするのが望ましいです。
